# Phase 2: Training CLI - Context

**Gathered:** 2026-04-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver `src/train.py` — a typer CLI that orchestrates the four RVC training stages (1: preprocess, 2: F0 extract, 3: feature extract, 4: train) end-to-end on a provisioned pod. The file mirrors `src/generate.py` architecturally: pure arg-builder helpers (`build_rvc_preprocess_cmd`, `build_rvc_extract_f0_cmd`, `build_rvc_extract_feature_cmd`, `build_rvc_train_cmd`) + doctor pre-flight + thin subprocess orchestrator. Also extend `src/doctor.py` with two new training-specific checks (`check_pretrained_v2_weights`, `check_training_dataset_nonempty`) so `python src/doctor.py --training` remains the single source of training-readiness truth. Also deliver `_write_filelist(exp_dir, version, sample_rate)` — the helper that reproduces `infer-web.py:click_train`'s filelist format and injects mute-reference rows.

**Out of scope for Phase 2:** Stage 5 FAISS index build (Phase 3), `models/<name>/` auto-export + manifest (Phase 3), `scripts/train.sh` shell wrapper (Phase 4), `--dataset-url` remote pull (Phase 4), `DONE`/`FAILED` sentinel files (Phase 4), README pod docs (Phase 5).

<decisions>
## Implementation Decisions

### CLI Surface — Presets + Granular Overrides
- **D-01:** `src/train.py` ships a `--preset` flag with four values: `smoke` / `low` / `balanced` / `high`. Default is `balanced`. The preset resolves to a dict of `epochs`, `batch_size`, `save_every`; any explicit flag (`--epochs`, `--batch-size`, `--save-every`) overrides the corresponding preset value via a simple `preset_dict.update(explicit_overrides)` pass after typer parsing.
- **D-02:** Preset values (authoritative; planner will codify as a dict constant in `src/train.py`):
  - `smoke`: `epochs=1, batch_size=1, save_every=1` — satisfies ROADMAP.md Phase 2 success criterion #1 (`--epochs 1 --batch-size 1` smoke run exits 0 or 61 with a `.pth` on disk).
  - `low`: `epochs=200, batch_size=6, save_every=50` — tuned for the 12 GB VRAM floor from Phase 1 (`check_gpu_vram_floor(12)`). Target cards: RTX 3060/4070-class.
  - `balanced` (DEFAULT): `epochs=300, batch_size=12, save_every=50` — tuned for 24 GB cards (RTX 3090/4090). This is the bare-`python src/train.py` assumption.
  - `high`: `epochs=500, batch_size=40, save_every=50` — tuned for H100/H200/A100-80GB class. Aggressive batch to use the headroom.
- **D-03:** `--preset` and explicit overrides are **mixable**. `python src/train.py --preset high --epochs 800` yields `epochs=800, batch_size=40, save_every=50`. Rationale: presets are starting points, not mutually exclusive regimes. Cleaner UX than forcing users to retype four flags to tweak one.
- **D-04:** Other flag defaults: `--sample-rate` default is `40000` (RVC community standard, middle tier, most pretrained_v2 weights ship at 40k). `--rvc-version` default is `v2` (matches `3_feature768/` probe path). `--f0-method` default is `rmvpe` (most accurate; CPU-or-GPU; no GPU contention during Stage 2 unlike `rmvpe_gpu`). All are explicit CLI flags per TRAIN-01; not hidden in a preset.
- **D-05:** `--resume` flag is **removed from the CLI**. Resume is **always-on probe-and-skip** (see D-08 below). This is a deliberate deviation from TRAIN-01's literal list of flags: the requirement's spirit is "resumable training", which D-08's intrinsic-probe design delivers more robustly than a user-toggled flag. Mirrors the Phase 1 "intrinsic-probe-only" philosophy (01-CONTEXT.md D-12). The planner must note this deviation in PLAN.md so verification does not fail on a missing `--resume` flag.

### Filelist Generation (TRAIN-05)
- **D-06:** `_write_filelist(exp_dir: Path, version: str, sample_rate: int) -> Path` is **format-equivalent** to `rvc/infer-web.py:click_train` lines 500-546 — not byte-exact. The unit test asserts: (a) every line has the expected pipe-field count, (b) path fields point at the real files in `0_gt_wavs/`, `2a_f0/`, `2b-f0nsf/`, `3_feature768/` (v2) or `3_feature256/` (v1), (c) sid values are all `0`, (d) one mute-reference row exists per mute file found under `rvc/logs/mute/`, (e) the resulting file is non-empty. This keeps tests stable against incidental RVC formatting quirks while guaranteeing RVC's `train.py` parser accepts the output.
- **D-07:** Mute-reference rows are generated by globbing `rvc/logs/mute/0_gt_wavs/*.wav` (or whatever directory pattern `click_train` uses) at helper-invocation time and injecting one row per file found. **No hard-coded count.** This keeps the row count driven by whatever `rvc/tools/download_models.py` shipped (currently 3), and makes the helper robust if upstream RVC changes the mute-ref asset set. Phase 1's `check_rvc_mute_refs` doctor check validates their presence as pre-flight — if zero are found, pre-flight fails before `_write_filelist` runs.

### Resume + Stage-Skip Probe Semantics
- **D-08:** Resume is **always-on intrinsic probe-and-skip**. Each stage's "already done" probe compares expected file count against the **input dataset file count** (`N = count of audio files in --dataset-dir matching AUDIO_EXTS from src/preprocess.py`):
  - **Stage 1 skip** iff `count(rvc/logs/<name>/0_gt_wavs/*.wav) >= N`.
  - **Stage 2 skip** iff `count(rvc/logs/<name>/2a_f0/*.npy) >= N` AND `count(rvc/logs/<name>/2b-f0nsf/*.npy) >= N`.
  - **Stage 3 skip** iff `count(rvc/logs/<name>/3_feature768/*.npy) >= N` for v2, or `3_feature256/*.npy` for v1.
  - On a file-count mismatch (stage crashed mid-way), the stage re-runs. RVC's preprocess/extract scripts overwrite per-file, so re-run is safe.
  - Log output on skip: `"Stage N: skipping — already populated (<count> files)"`. Log output on match-fail: `"Stage N: re-running — expected >= <N> files, found <count>"`.
- **D-09:** **Stage 4 (train) resume** defers entirely to RVC's built-in checkpoint resume. RVC's `train.py` auto-loads the latest `G_*.pth` / `D_*.pth` in `rvc/logs/<name>/` when re-invoked — we do nothing special. **Fast-path:** if `rvc/assets/weights/<name>.pth` already exists and is non-empty (≥ 1 KiB), skip Stage 4 entirely with log `"Stage 4: skipping — final weight already present"`. This makes a fully-complete run a no-op on re-invocation.
- **D-10:** **Experiment collision handling** follows the same probe-and-continue philosophy. No `--force`, no timestamped run dirs, no prompts. Same experiment name + same dataset = resume. Same name + different dataset = file-count mismatch forces the mismatched stages to re-run. The user is responsible for picking a distinct `--experiment-name` if they want a clean slate.

### Doctor Pre-flight (TRAIN-06)
- **D-11:** All TRAIN-06 checks live in `src/doctor.py` and are composed into the `--training` flag's check set alongside the Phase 1 additions. New check functions added in this phase:
  - `check_pretrained_v2_weights(sample_rate: int, version: str, if_f0: bool) -> CheckResult` — verifies the exact set of files under `rvc/assets/pretrained_v2/` (or `pretrained/` for v1) that `train.py` will load as `-pg`/`-pd`, given the sample rate, version, and f0 flag. Returns fail with a concrete fix hint if any are missing (e.g., `"Run scripts/setup_rvc.sh to download pretrained_v2/f0G40k.pth"`). Prevents the "silent random-init training" pitfall in STATE.md.
  - `check_training_dataset_nonempty(dataset_dir: Path) -> CheckResult` — verifies `dataset_dir` exists, is a directory, and contains at least one file matching `AUDIO_EXTS` (re-use the tuple from `src/preprocess.py`).
- **D-12:** `check_hubert_base` and `check_rvc_mute_refs` were already plumbed in Phase 1's `--training` composition per 01-CONTEXT.md D-09. Phase 2 consumes them; it does **not** re-define them. If Phase 1's plumbing landed as stubs instead of full implementations, Phase 2's planner must identify this gap and include the full bodies as part of planning.
- **D-13:** `src/train.py` calls the full training check set via a single doctor helper (e.g., `doctor.run_training_checks(dataset_dir=..., sample_rate=..., version=..., if_f0=True)` returning `list[CheckResult]`) before any subprocess runs. On any fail, print a `rich.Table` summary and exit 1 with the first fix_hint. Matches `src/generate.py`'s pre-flight pattern.

### Subprocess Logging + Output Visibility
- **D-14:** Training subprocesses are **streamed live to the terminal AND tee'd to `rvc/logs/<name>/train.log`**. Implementation: `subprocess.Popen(..., stdout=subprocess.PIPE, stderr=subprocess.STDOUT, bufsize=1, text=True, cwd=RVC_DIR, env=...)`, then iterate `popen.stdout` line-by-line writing each line to both `sys.stdout` and the opened log file handle. No `capture_output=True`, no waiting-then-printing. This deviates from `src/generate.py`'s buffered approach because training stages run for hours, not seconds — a billing-pod user must see liveness in real time, and SSH drops must not lose the post-mortem log.
- **D-15:** On subprocess non-zero exit (and the exit code is not 0 or 61 per TRAIN-07), `_tail` reads the last 30 lines of `rvc/logs/<name>/train.log` (not a Python buffer) and prints them with pipeline-stage context prefix. This handles multi-GB cumulative stdout without RSS blowup. The `_tail` helper itself mirrors `src/generate.py:_tail` but reads from a file path instead of a string.
- **D-16:** Per-stage log files: each stage writes to the **same** `rvc/logs/<name>/train.log`, appended, with a stage-header banner line (`"===== Stage N: <name> (started HH:MM:SS) ====="`) before each subprocess. Rationale: one log file per experiment is simpler than four, and a resumed run's log grows linearly instead of fragmenting.

### Verification + Exit Codes
- **D-17:** Exit codes follow `src/generate.py` convention (TRAIN-12):
  - `0` success.
  - `1` config/setup error: missing weights, missing dataset dir, failing doctor pre-flight.
  - `2` user-input error: invalid CLI flag combinations (bad preset name, bad sample rate, bad f0 method, unknown rvc version).
  - `3` subprocess runtime error: Stage N subprocess exited with a non-{0,61} code.
- **D-18:** After Stage 4 completes, `src/train.py` cross-checks that `rvc/assets/weights/<name>.pth` exists and is non-empty (≥ 1 KiB floor, matching the `generate.py` output check). If missing despite a 0/61 exit, exit 3 with the log tail and a hint (`"RVC reported success but no weight file was produced — check train.log for silent failures"`).
- **D-19:** Subprocess environment for every stage: the current `os.environ` copy, plus `TRANSFORMERS_OFFLINE=1`, `HF_DATASETS_OFFLINE=1`, `LANG=C.UTF-8` (TRAIN-14). Set once in a helper, passed to every `Popen` call.

### Arg Builders (TRAIN-03, TRAIN-04)
- **D-20:** Four pure arg-builder functions in `src/train.py`, following the `build_rvc_subprocess_cmd` pattern from `src/generate.py`. Each takes keyword-only arguments, returns `list[str]`, has zero side effects, and is fully unit-tested without a GPU. Signatures (authoritative shapes; planner may refine parameter order):
  - `build_rvc_preprocess_cmd(*, rvc_python: Path, dataset_dir: Path, sample_rate: int, num_procs: int, exp_name: str) -> list[str]` — mirrors `infer-web.py:preprocess_dataset` argv.
  - `build_rvc_extract_f0_cmd(*, rvc_python: Path, exp_name: str, num_procs: int, f0_method: str) -> list[str]` — mirrors `extract_f0_print.py` argv.
  - `build_rvc_extract_feature_cmd(*, rvc_python: Path, exp_name: str, version: str, num_procs_per_gpu: int) -> list[str]` — mirrors `extract_feature_print.py` argv.
  - `build_rvc_train_cmd(*, rvc_python: Path, exp_name: str, sample_rate: int, version: str, epochs: int, batch_size: int, save_every: int, f0_method: str, pretrained_g: Path, pretrained_d: Path, if_f0: bool, gpus: str) -> list[str]` — mirrors `train_nsf_sim_cache_sid_load_pretrain.py` argv as invoked by `click_train`.
- **D-21:** `build_rvc_train_cmd` ALWAYS passes `-pg` and `-pd` explicitly with absolute paths resolved from the sample rate + version, even though they have defaults in RVC's argparse. This closes the STATE.md pitfall: "Missing pretrained weights cause silent random-init training — always pass -pg/-pd explicitly."
- **D-22:** Unit tests in `tests/unit/test_train.py` assert exact argv contents for each builder for representative inputs (preset=`smoke`, preset=`low`, v1 vs v2, different f0 methods, different sample rates). **No GPU required.** The builders are pure data transformations, so tests run in the normal `.venv` pytest.

### Sample-rate Consistency (TRAIN-13)
- **D-23:** The CLI's `--sample-rate` value is passed into **both** `build_rvc_preprocess_cmd` (as the resample target) and `build_rvc_train_cmd` (as the `-sr` flag), from the same single variable in `main()`. No chain of intermediate variables; no config file. Unit tests cover both builders with the same sample rate value and assert both carry it. Closes the STATE.md pitfall: "Sample-rate chain mismatch between stages."

### Two-Venv Boundary (TRAIN-10)
- **D-24:** `src/train.py` **must not** contain `import torch`, `import fairseq`, `import faiss`, or `import rvc` anywhere. All RVC interaction is via `subprocess.Popen` with `cwd=RVC_DIR` and `shell=False`. A ruff-level check (or a simple `grep -E "^import (torch|fairseq|faiss)" src/train.py` in the Phase 2 test suite) enforces this. Matches Phase 1's absolute two-venv boundary.

### Claude's Discretion
- Exact typer `Option(...)` help text, `rich.Table` cosmetic formatting, internal helper naming inside `src/train.py` and `src/doctor.py`, and log banner wording — planner and executor pick what reads best.
- Exact preset dict structure (nested dict vs dataclass vs dict-of-dataclasses). D-02 names the values; the container shape is flexible.
- Per-stage `num_procs` / `num_procs_per_gpu` defaults — use RVC's historical defaults (typically `num_procs = os.cpu_count()` capped reasonably) unless a concrete problem emerges in testing.
- `gpus` argv value for `build_rvc_train_cmd` — default to `"0"` (single-GPU target per project constraints).
- Whether `_write_filelist` lives in `src/train.py` or a new `src/train_filelist.py` helper module — inline is fine since it's one function.
- The stage-header banner wording in the shared log file (D-16).

<specifics>
## Specific Ideas

- **Smoke-test invocation** (from ROADMAP success criterion #1) is equivalent to `--preset smoke --experiment-name smoke --dataset-dir dataset/processed/`. The planner should verify this explicit command line lines up byte-for-byte with the ROADMAP text (which passes `--epochs 1 --batch-size 1` explicitly); D-01/D-02/D-03 support both the explicit-flag and `--preset smoke` form.
- **Smoke run is the ONLY test that touches real RVC stages in the unit-test tier.** It's marked with `@pytest.mark.gpu` and excluded from the default `pytest` invocation (project convention). CI-safe unit tests exercise only the builders and `_write_filelist`.
- **Log tail pattern** (D-15): if `rvc/logs/<name>/train.log` does not exist at failure time (stage crashed before a single line was written — rare), fall back to the `_tail` of the subprocess's `stderr` pipe. Defensive, cheap.
- **`num_procs` default** for the preprocess builder is probably `min(os.cpu_count() or 1, 8)`. RVC historically uses `n_p` in that ballpark. Planner should verify against `click_train`'s default and align.
- **Progress signal**: RVC's own train.py writes TensorBoard events into `rvc/logs/<name>/`. We don't read them (no `import torch`); the user running `tail -f rvc/logs/<name>/train.log` is the liveness UX.
- **`.env` is NOT loaded by `src/train.py`.** `src/generate.py` loads `.env` for `DEFAULT_EDGE_VOICE` and `DEFAULT_MODEL`; `src/train.py` has no such per-user config keys. Keep it deterministic — every flag is a CLI flag.

</specifics>
